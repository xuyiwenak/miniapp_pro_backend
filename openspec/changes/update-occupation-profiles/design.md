# Design Document: 职业画像参数优化

## Context

当前职业数据库包含37个职业，每个职业定义了 Big Five 五大人格维度的要求。然而，通过实际测评发现三个问题：

1. **评分范围问题**：职业要求范围（-0.1 ~ 0.7）远低于用户实际分数范围（-3 ~ +3），导致高分用户（O=1.85, N=1.41）被认为"过犹不及"，匹配分数偏低（35.4分）。
2. **维度使用问题**：78%的职业只用3维（O, C, N），忽略外向性（E）和宜人性（A）对社交型、共情型职业的重要性。
3. **安全性问题**：仅11%的职业有硬性门槛，可能推荐不适合的高风险职业给能力不足的用户。

### 现有缓解措施

已通过"方向性匹配"算法缓解问题1（高分用户被惩罚）：
```typescript
// 如果用户特质高于职业要求，只按50%差距计算
if (oDiff > 0) oDiff *= 0.5;
if (cDiff > 0) cDiff *= 0.5;
if (nDiff > 0) nDiff *= 0.5;
```

但这是**治标不治本**，应该从源头优化职业画像参数。

## Goals / Non-Goals

### Goals
1. ✅ 扩大职业评分范围，减少对高分用户的不公平惩罚
2. ✅ 为社交型职业启用外向性维度，提升匹配精度
3. ✅ 为共情型职业启用宜人性维度，筛选合适候选人
4. ✅ 为高风险职业设置硬性门槛，保障推荐安全性
5. ✅ 保持数据可维护性，记录调整理由

### Non-Goals
- ❌ 基于真实从业者数据（需长期调研，不在本次范围）
- ❌ 机器学习优化参数（需大量训练数据，未来考虑）
- ❌ 动态调整参数（暂不支持，需架构变更）

## Decisions

### Decision 1: 评分范围扩大策略

**选择**: 根据职业类型分级扩大

| 职业类型 | 当前范围 | 新范围 | 示例 |
|---------|---------|--------|------|
| 创意型（高开放性） | O: 0.4-0.7 | O: 0.8-1.5 | 艺术总监、科研人员、游戏设计师 |
| 高压型（高稳定性） | N: 0.3-0.5 | N: 0.6-1.0 | 临床医生、投资分析师、律师 |
| 严谨型（高尽责性） | C: 0.5-0.6 | C: 0.7-1.0 | 财务会计、审计师、企业法务 |
| 普通型 | 各维度 0-0.4 | 各维度 0-0.8 | 其他职业 |

**理由**:
- 参考 Big Five 理论：标准分的正常范围是 -3 ~ +3，68%的人在 -1 ~ +1
- 优秀从业者通常在 +1 ~ +2 范围（84th ~ 98th percentile）
- 职业要求应该覆盖到这个范围，而非局限于 0 ~ 0.7

**Alternatives considered**:
- 方案A：统一提升所有职业（简单但不精确）❌
- 方案B：按职业类型分级调整（精确但工作量大）✅ **采纳**
- 方案C：保持现状，依赖方向性匹配（治标不治本）❌

### Decision 2: 外向性（E）启用策略

**选择**: 为需要社交、沟通、表达的职业启用外向性维度

| 职业类别 | E 范围 | 示例 |
|---------|-------|------|
| 高社交（销售、市场） | E: 0.3-0.5 | 销售经理、市场营销经理、保险规划 |
| 中社交（教育、管理） | E: 0.2-0.3 | 学科教师、人力资源经理、企业培训师 |
| 低社交（设计、创作） | E: 0.0-0.2 | 品牌设计师、艺术总监、UX设计师 |
| 负社交（科研、技术） | E: -0.2-0.0 | 科研人员（已有）、游戏设计师（已有） |

**理由**:
- 外向性影响职业绩效：Judge et al. (2002) 发现外向性与销售、管理岗位绩效显著正相关
- 当前仅19%职业使用外向性，导致无法区分"需要社交"和"不需要社交"的职业
- 销售经理的 E=0.5 已被验证合理，应扩展到类似职业

**Alternatives considered**:
- 方案A：所有职业都加外向性（不现实，科研人员不需要）❌
- 方案B：只加社交型职业（精准且高效）✅ **采纳**
- 方案C：保持现状（错失匹配精度提升）❌

### Decision 3: 宜人性（A）启用策略

**选择**: 为需要共情、关怀、协作的职业启用宜人性维度

| 职业类别 | A 范围 | 示例 |
|---------|-------|------|
| 高共情（医疗、社工） | A: 0.4-0.5 | 心理咨询师、社会工作者、护理师 |
| 中共情（教育、HR） | A: 0.3-0.4 | 学科教师、人力资源经理、企业培训师 |
| 低共情（设计、产品） | A: 0.2-0.3 | UX设计师、产品经理 |

**理由**:
- 宜人性与助人职业绩效正相关：Mount et al. (1998)
- 当前仅8%职业使用宜人性，导致无法筛选出真正适合助人职业的候选人
- 护理师的 A=0.5 已被验证合理，应扩展到类似职业

**Alternatives considered**:
- 方案A：所有职业都加宜人性（不合理，竞争型岗位不需要）❌
- 方案B：只加助人/协作型职业（精准）✅ **采纳**
- 方案C：保持现状（安全隐患，可能推荐低共情者做咨询师）❌

### Decision 4: 硬性门槛设置策略

**选择**: 为高风险、高要求职业设置最低门槛

| 职业类别 | 门槛 | 理由 |
|---------|------|------|
| 高压医疗 | N ≥ 0.3-0.4, A ≥ 0.2-0.3 | 需要情绪稳定和患者关怀 |
| 高压金融 | N ≥ 0.3, C ≥ 0.3 | 需要抗压和严谨 |
| 科研/教育 | O ≥ 0.2-0.3, A ≥ 0.1-0.2 | 需要好奇心和教学能力 |
| 销售/咨询 | E ≥ 0.2, N ≥ 0.1 | 需要社交能力和抗压 |

**理由**:
- 安全性优先：不适合的推荐可能导致职业失败或伤害他人（如医生情绪崩溃）
- 参考已有案例：律师、审计师、护理师已有门槛，效果良好
- 门槛值设置：通常为要求值的 50-70%（如要求 N=0.5，门槛 N=0.3）

**Alternatives considered**:
- 方案A：所有职业都加门槛（过度限制，普通职业不需要）❌
- 方案B：只加高风险职业（安全且合理）✅ **采纳**
- 方案C：保持现状（安全隐患）❌

### Decision 5: 参数调整方法

**选择**: 基于文献、专家经验和已有数据的混合方法

**步骤**:
1. 查阅 Big Five 与职业绩效的元分析文献（Barrick & Mount, 1991; Judge et al., 2002）
2. 参考已有的合理配置（销售经理 E=0.5, 护理师 A=0.5）
3. 按职业类别批量调整，保持内部一致性
4. 设置验证脚本检测异常值

**理由**:
- 缺乏真实数据时，文献和专家经验是次优选择
- 已有8个5维职业的配置可作为参照
- 批量调整效率高，内部一致性好

**Alternatives considered**:
- 方案A：随机猜测（不可接受）❌
- 方案B：混合方法（当前最优）✅ **采纳**
- 方案C：等待真实数据（时间成本高，短期无法实施）❌

## Risks / Trade-offs

### Risk 1: 现有用户结果变化

**风险**: 已完成测评的用户重新计算后，推荐职业可能改变。

**影响**: 用户可能困惑："为什么上次推荐 A 职业，这次推荐 B？"

**缓解**:
- 方案A：保留旧版本数据，支持 A/B 测试
- 方案B：在用户界面显示"算法已优化，建议重新查看推荐"
- 方案C：仅对新用户生效，老用户需手动触发重新计算

**选择**: 方案A + 方案B（最大灵活性）

### Risk 2: 参数校准错误

**风险**: 手动调整37个职业 × 5个维度 = 185个参数，容易出错。

**影响**: 错误的参数可能导致匹配结果不合理（如程序员要求 E=0.8）。

**缓解**:
- 自动化验证脚本检测：
  - 异常值（> 2.0 或 < -1.0）
  - 矛盾配置（创意职业 O < 0.2）
  - 门槛值超过要求值（minimumReq.N > required.N）
- Code review：人工检查关键职业的配置
- 分批发布：先优化10个核心职业，观察1周效果

**选择**: 全部实施

### Risk 3: 缺乏真实数据验证

**风险**: 基于文献和经验估计，未必符合中国职场实际。

**影响**: 可能存在系统性偏差（如低估了某些职业的开放性要求）。

**缓解**:
- 短期：通过用户反馈迭代（"您觉得这个推荐准确吗？"）
- 中期：收集实际从业者数据，反向优化参数
- 长期：机器学习自动优化（需积累足够训练数据）

**接受度**: 这是当前条件下的最优解，可接受

## Migration Plan

### Phase 1: 准备阶段（已完成）
- ✅ 分析现有数据
- ✅ 生成诊断报告
- ✅ 创建 OpenSpec 提案

### Phase 2: 实施阶段（进行中）
1. 备份原始数据：`cp tpl/seed_occupation.json tpl/seed_occupation.json.backup`
2. 按优先级批量修改：
   - 高优先级：扩大评分范围 + 增加门槛（预计 3小时）
   - 中优先级：启用 E/A 维度（预计 2小时）
3. 运行验证脚本：`npx tsx test/check_occupation_dimensions.ts`
4. 修复异常值和错误
5. 生成对比报告

### Phase 3: 测试阶段
1. 使用之前的测试用例重新运行匹配
2. 对比修改前后的分数变化
3. 验证高分用户（O=1.85, N=1.41）的分数是否提升
4. 检查硬性门槛是否正确过滤

### Phase 4: 发布和监控
1. 更新 matching_explain.json 版本号
2. 部署到生产环境
3. 监控用户反馈和匹配分数分布
4. 必要时微调参数

### Rollback Plan
如果发现严重问题（如所有用户分数异常）：
```bash
# 回滚职业数据
cp tpl/seed_occupation.json.backup tpl/seed_occupation.json

# 重启服务
pm2 restart all
```
