**内容管理**

**技术支撑**

- OSS 存储

### 2. 核心缺口发现

> ⚠️ **分析是单次的、孤立的**

每次分析都是一张作品的快照，彼此没有连接。用户无法看到：
- 「我这段时间情绪在往哪走」
- 「我创作风格的变化」
- 「我在慢慢变好吗」

这是一个系统性的缺口，影响的不只是某个功能，而是产品的「疗愈感」本身——
没有时间维度，就没有「疗愈」，只有「分析」。

### 3. 扩展方向初步分类

按价值维度分为四个象限：

```
                  高留存
                    │
    纵向深化        │         横向扩展
    （时间轴）       │         （新功能）
                    │
  ──────────────────┼──────────────────
                    │
    社群与传播       │         变现转化
    （低优先）       │         （课程推荐）
                    │
                  低留存
```

**A. 纵向深化（时间轴方向）**
- 情绪轨迹图：跨作品追踪8维度随时间的变化
- 疗愈进度报告：周/月维度的叙事性情绪总结
- 多作品对比：两幅作品并排比较情绪差异
- MBTI × 情绪关联：性格类型与创作情绪的长期画像

**B. 横向扩展（新功能方向）**
- AI 疗愈对话：基于报告与 AI 进行一对一深度对话
- 作品系列：将多幅作品归组为一个「情绪系列」
- 色彩推荐引擎：根据情绪状态推荐下次的创作色彩
- 情绪日记：图文双输入联合分析

**C. 变现转化**
- 课程推荐：根据情绪分析匹配工作室课程
- PDF 专业报告：可打印的治疗记录（供心理咨询使用）

**D. 社群与传播**
- 情绪共鸣互动：公开作品上的情绪化互动（非点赞）
- 情绪地图：所有用户情绪的匿名可视化

### 4. 待厘清的核心问题

在进入下一步设计之前，以下问题需要明确答案，因为它们会影响整个扩展逻辑：

**Q1：用户定位**
> Mandis 的用户现在主要是在「有趣的 AI 体验」阶段，还是真的在用它做持续的自我疗愈？
> 这两种定位，产品演进方向完全不同。

**Q2：分析受众**
> 分析报告是只给用户自己看，还是有「第三方」场景？
> 比如：艺术治疗师、工作室老师、家长看孩子作品？

**Q3：扩展驱动力**
> 留存 / 变现 / 深度 / 传播 ——当前阶段优先级是什么？

---

### 5. 三个问题的答案

**A1：用户定位 → 专业疗愈工具**

不是娱乐，不是体验。Mandis 要做的是一个专业的疗愈评估工具，功能深度和语言风格都应匹配专业场景。这意味着：
- 分析报告需要有临床参考价值，而不只是「好看的雷达图」
- 语言体系要偏向心理/疗愈领域的专业表达
- 用户对功能的期待是「有用」，而不是「有趣」

**A2：分析受众 → 疗愈师 + 家长（三方场景）**

报告不只是给用户自己看的：
- 成人用户：本人 + 疗愈师
- 儿童用户：家长 + 疗愈师（儿童本人不具备阅读报告的能力）

这意味着系统需要支持「报告分享」或「疗愈师查看」的机制。

**A3：扩展驱动力 → 变现（推课程、推服务）**

留存不是核心目标——这是一个按次交互的小程序，用户不需要每天打开。
核心目标是：**分析结果能转化为课程或服务的购买**。

另：工作室有现成的服务号，可作为推广和引流的渠道，目前缺乏完整的转化方案。

---

### 6. 产品真实形态（重新定义）

根据以上答案，产品不是「用户自助分析工具」，而是一个**疗愈评估 + 服务推荐漏斗**：

```
  ┌──────────────────────────────────────────────────────────┐
  │                   MANDIS 产品漏斗                         │
  └──────────────────────────────────────────────────────────┘

  [引流]           [评估]          [推荐]          [转化]
  服务号推文  ──▶  上传作品  ──▶  专业疗愈报告  ──▶  课程 / 服务
                   ↓                  ↓
                 AI 分析         ┌────┴────┐
                              疗愈师    家长/本人
                              读报告    读报告
                                  ↓
                              发现情绪需求
                                  ↓
                              推荐课程 / 服务
                                  ↓
                              服务号跟进 + 转化
```

### 7. 新增的待厘清问题

产品定位明确后，以下三个执行层问题需要在下一步推演中解决：

**Q4：疗愈师角色模型**

疗愈师是怎么进入这个系统的？三种可能：

```
模型 A — 疗愈师有账号，主动查看客户报告
┌────────┐  分配客户   ┌────────┐  查看报告  ┌────────┐
│工作室  │ ──────────▶ │疗愈师  │ ─────────▶ │报告详情│
└────────┘             └────────┘             └────────┘

模型 B — 用户/家长主动分享报告链接给疗愈师
┌────────┐  生成链接   ┌────────┐  打开链接  ┌────────┐
│用户    │ ──────────▶ │分享    │ ─────────▶ │疗愈师  │
└────────┘             └────────┘             └────────┘

模型 C — 疗愈师不参与系统，系统自动推荐课程
                用户分析 ──▶ 系统自动推荐课程
                （疗愈师在课程里，不在系统里）
```

> 疗愈师有没有「管理客户」的需求？还是主要在线下接触？

**Q5：课程和服务的落地形态**

- 工作室现在是否有课程列表（线上/线下）？
- 推荐后用户如何报名——小程序页面、服务号、直接联系老师？
- 「推荐」是 AI 自动推，还是疗愈师手动给建议？

**Q6：服务号的当前状态**

- 服务号现在是纯推文，还是有菜单 / 客服对话功能？
- 是否已与小程序关联（同一主体）？
- 能否发模板消息（用户关注后接收通知）？

---

### 8. 执行层问题的答案（Q4-Q6）

**A4：疗愈师角色模型 → 模型 A（疗愈师有账号，主动管理客户）**

- 疗愈师通过 art_web 门户网站登录，管理自己的客户
- 小程序用户绑定手机号后，art_web 可识别该用户（同一账号体系）
- 疗愈师需要能在 art_web 查看客户的疗愈分析报告

**A5：课程和服务的落地形态**

- 工作室已有线下课程（目前未上线）
- 推荐路径：小程序测评 → 拿报告结果 → 疗愈师推荐具体课程和疗愈方案
- 小程序是「入口 + 评估」，art_web 是「疗愈师工作台 + 推荐」
- 课程需要从静态展示升级为数据库实体，支持「推荐绑定」

**A6：服务号定位 → 高端用户入口 + 内容运营 + 客服**

- 服务号作为小程序测评的入口
- 推文做内容种草和引流
- 客服对话承接咨询需求
- 目标用户群体：相对高端消费人群，体验质量要求高
- 服务号与小程序需要清晰的功能对接（跳转、通知、绑定）

### 10. 当前系统缺口（对照架构）

通过查看 art_web 代码，现有管理后台页面：
`Dashboard / Users / Works / Feedback / System`

**缺失的关键功能：**

| 缺口 | 影响 | 优先级 |
|------|------|--------|
| 无「疗愈师」角色 | 疗愈师只能用管理员身份登录，权限边界模糊 | 高 |
| 无客户报告查看页 | 疗愈师无法在 art_web 查看用户的疗愈报告 | 高 |
| 课程是静态硬编码 | 无法做「推荐绑定」，课程不可管理 | 高 |
| 无推荐记录模型 | 疗愈师对客户的推荐没有任何持久化 | 中 |
| 无服务号通知对接 | 推荐结果无法主动触达用户 | 中 |
| 无手机号登录（疗愈师端）| 疗愈师登录体验待确认 | 待确认 |

**推荐的两种模式（待决策）：**
- **AI 自动推荐**：基于情绪分析结果自动匹配课程标签，疗愈师确认后发送
- **疗愈师手动推荐**：疗愈师阅读报告后自由选择课程和方案，附加个人留言
